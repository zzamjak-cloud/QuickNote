## 데이터 보존

### 현황

Phase 6/7에서 persist 마이그레이션 회귀 테스트·outbox 통합 테스트가 완료됐으나,
런타임 데이터 손실 방어(quarantine 복구·dead letter 관리·부분 실패 안전화)는 인프라만 있고
사용자 가시성·자동 정리·CI 게이트가 없는 상태.

| # | 기준 | 검증 |
|---|------|------|
| AD1 | migrationQuarantine 항목이 0이 아니면 DevTools 패널에 경고 표시 + 자동 복구 재시도 1회 | devtools snapshot 수동 확인 |
| AD2 | outbox dead letter 수를 WorkspaceSyncBanner 에 표시; 30일 경과 항목 자동 만료 | engine.test 에 TTL 케이스 추가 |
| AD3 | Bootstrap fetchApply 중 pages/databases/comments 개별 실패 허용; 성공한 도메인만 적용, 기존 캐시 유지 | Bootstrap.test 부분 실패 시나리오 3건 |
| AD4 | permanentTombstones 30일 초과 항목을 앱 시작 시 자동 정리; legacyCleanup에 통합 | 단위 테스트로 TTL 경계 검증 |
| AD5 | DATA_KEYS 배열 길이 변경 시 PERSIST_VERSION 미변경이면 테스트 fail (CI gate) | migrations.test.ts 에 version-data-keys 연동 케이스 |

### Phase D1 — migrationQuarantine 가시성 + 자동 복구 (1~2일)

**현재**: `databaseStore.migrationQuarantine[]` 에 복구 불가 원본이 쌓이지만 노출 없음.

1. `src/lib/devtools/snapshot.ts` 에 `quarantineReport()` 헬퍼 추가
2. DevTools 콘솔 커맨드(`window.__qn.quarantine()`) 로 항목 조회 가능
3. `attachPersistedMeta` / `mergePersistedSubset` 을 실행한 뒤 복구 가능하면 자동 재시도 1회 (store init 시)
4. 복구 성공 시 quarantine 에서 제거; 실패 시 경고 console.error

- **AC**: AD1
- **파일**: `src/lib/migrations/persistedStore.ts`, `src/lib/devtools/snapshot.ts`, `src/store/databaseStore.ts`

### Phase D2 — outbox dead letter 관리 (1일)

**현재**: dead letter 항목(payload-too-large / resource-already-gone)이 별도 저장되지만 사용자에게 노출 안 됨.

1. `OutboxAdapter` 인터페이스에 `listDeadLetters(limit)` / `clearDeadLetters()` 추가
2. `adapter.web.ts` / `adapter.tauri.ts` 구현 (IndexedDB `deadLetters` 오브젝트 스토어)
3. `WorkspaceSyncBanner.tsx` 에 dead letter 수 뱃지 + "오류 항목 제거" 버튼
4. `SyncEngine.flush` 에 30일 경과 dead letter 자동 만료 로직 추가

- **AC**: AD2
- **파일**: `src/lib/sync/outbox/types.ts`, `adapter.web.ts`, `adapter.tauri.ts`, `src/components/sync/WorkspaceSyncBanner.tsx`

### Phase D3 — Bootstrap 부분 실패 안전화 (1일)

**현재**: `fetchApply` 에서 pages/databases/comments 중 하나라도 throw 하면 전체 재페치가 실패하고 화면이 멈춤.

1. `fetchApply` 를 `Promise.allSettled` 기반으로 교체; 각 도메인 독립 적용
2. 실패한 도메인은 기존 캐시 유지 (`workspaceLoading` 은 성공 도메인만으로 진행)
3. 부분 실패 시 `WorkspaceSyncBanner` 에 "일부 데이터 로드 실패" 경고 표시
4. `Bootstrap.test.ts` 에 부분 실패 3 시나리오(pages만 실패 / databases만 실패 / 전체 실패) 추가

- **AC**: AD3
- **파일**: `src/Bootstrap.tsx`, `src/__tests__/Bootstrap.test.tsx`

### Phase D4 — localDeleteGuards TTL 자동 정리 (0.5일)

**현재**: `permanentTombstones` 가 무한 누적됨. 대형 워크스페이스에서 메모리·localStorage 낭비.

1. `markPermanentlyDeletedEntity` 호출 시 `deletedAt: Date.now()` 함께 저장
2. 앱 시작(`legacyCleanup.ts` init) 시 30일 초과 tombstone 자동 삭제
3. 단위 테스트: 29일·30일·31일 경계 케이스

- **AC**: AD4
- **파일**: `src/lib/sync/localDeleteGuards.ts`, `src/lib/sync/legacyCleanup.ts`

### Phase D5 — persist version CI gate (0.5일)

**현재**: `DATA_KEYS` 배열이 바뀌어도 `PERSIST_VERSION` 을 올리지 않으면 조용히 통과.

- **AC**: AD5
